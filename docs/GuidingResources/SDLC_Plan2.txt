SDLC Plan — SPA-CGAL / Swept Path Analysis                                                                                 
  
 Context

 The user has a working CGAL-based C++ prototype that proves the swept path concept: given a 3D mesh and a sequence of 4×4  
 affine transforms, it unions transformed copies to produce a Swept Path Model (SPM). The long-term target is a real-time   
 UE5 component that animators can attach to any actor to visualize the swept volume across a configurable time window as    
 they adjust keyframes.

 The user wants to apply a formal 15-stage SDLC to this project, with document artifacts produced before significant        
 production code is written. The existing code is treated as a proof-of-concept spike, not production code. All artifacts   
 live in the repo (docs/ in Markdown + Mermaid/PlantUML), with a parallel export path to PDF/HTML for stakeholders.

 Key rationale

 - Stage 4 added (V&V Strategy early): Defining how you'll validate a requirement is a design activity, not an
 afterthought. For SPA, the acceptance criteria for "real-time" performance need to be established before architecture      
 choices are made — those criteria will directly determine whether CGAL is viable at all.
 - Stage 9 renamed: UML/SysML diagrams are produced during stages 6–8, not as a separate step afterward. Stage 9 is now a   
 synthesis gate — review, reconcile, and formally baseline the artifacts from 5–8 into a coherent design package before     
 implementation begins.
 - Stage 11 adds "Architectural Spike": The biggest technical risk is whether CGAL mesh boolean ops can meet real-time      
 performance targets inside UE5. Stage 11 must include a focused spike to answer this before full feature development.      
 - Stage 13 generalized: "Feedback loops" is UE5-integration-specific language. The broader label covers both the UE5       
 integration and any future integrations without changing the intent.
 - Risk management is omitted as a discrete stage — it is a continuous activity beginning at Stage 3. Each stage document   
 should include a risk delta section noting new risks identified and mitigations decided.

 Execution Approach

 We work through Stages 1–10 sequentially, one stage per conversation turn or small cluster of turns. At each stage:        

 1. Draft the artifact collaboratively (Claude writes, user reviews and refines)
 2. Commit the document to the repo
 3. Identify any risks uncovered and add them to the risk section of that document
 4. Only move to the next stage once the current artifact is baselined

 Starting point: Stage 1 — Mission, Vision & Success Criteria

 ---
 Critical Risk to Surface Early (Stage 3/4)

 The most significant architectural risk in the project is whether CGAL corefinement boolean ops can meet real-time
 performance targets inside UE5. The current spike shows:
 - 9 transform steps on a cylinder → sweep_result.obj at 6608 lines (manageable)
 - Performance for a production mesh with 60+ fps update rate is unknown

 This risk must be formally captured in Stage 3 (requirements) and quantified in Stage 4 (acceptance criteria) before the   
 architecture can be decided in Stage 8. Options that may emerge:
 - CGAL with aggressive optimization (parallel, incremental, level-of-detail)
 - UE5 Geometry Scripting (native, no CGAL dependency in UE)
 - SDF / voxel-based CSG (faster but approximate)
 - Convex hull / AABB approximation (fastest, least accurate)

 The Stage 11 architectural spike will test the leading candidate against the Stage 4 acceptance criteria.

 ---
 Verification

 Each stage is complete when:
 - The artifact document is committed to docs/
 - All required sections are populated (no TBDs on critical fields)
 - Any risks identified are documented with a mitigation note
 - The user has reviewed and approved the content

 End-to-end success criteria (established fully in Stage 4):
 - NFR acceptance tests pass for performance, accuracy, and UE5 integration
 - All FR-xxx items are implemented and validated
 - The UE5 component builds, attaches to a test actor, and updates in real-time within the defined time budget
